# 2026-03-03 — Dashboard Cron Investigation

## Context
User provided the OpenClaw cron definition for `dashboard-data-updater`:
```
python3 /home/ubuntu/.openclaw/workspace/scripts/refresh-dashboard-data.py >> /home/ubuntu/.openclaw/workspace/memory/dashboard-data.log 2>&1
```

## Discovery
- OpenClaw cron job `dashboard-data-updater` (ID: 4feaa4ea-0f1b-44d1-9ed0-50a2e531f442) is enabled and scheduled every 5 minutes in Asia/Bangkok.
- However, the Python script `refresh-dashboard-data.py` produces an outdated data structure:
  - `disk_history` is always empty `[]` (frontend expects 24-value sparkline)
  - Agent status fields mismatch: uses `sessionKey`, `kinds`, `activeSec` instead of `name`, `active`, `last_run`, `last_line`
- As a result, when this job runs, it overwrites `apps/dashboard/data.json` with incomplete data, breaking the dashboard's disk history and agent displays.

- There is a **duplicate system crontab entry** (`crontab -l`) that runs `./scripts/generate-dashboard-data.sh` every 5 minutes (UTC). This shell script:
  - Generates correct data structure including populated `disk_history` (persisted across runs via `memory/disk-history.json`)
  - Uses correct agent field names
  - Additionally performs git commit, push, and Vercel production deployment
  - Redirects output to `/dev/null`, so no explicit log
- The system crontab entry is actively updating `data.json` and appears to be the de facto producer of dashboard data.
- The two mechanisms conflict: whichever runs last determines the content. Since the system crontab uses the superior script, the dashboard currently works, but the OpenClaw cron's Python runs occasionally and corrupts the data (empty disk_history) until the next system crontab run.

## Issues
1. **Data corruption risk**: OpenClaw cron's Python script overwrites correct data with incomplete structure.
2. **Duplication**: Two independent cron systems are both trying to do the same job, causing thrashing.
3. **Documentation mismatch**: CRON_JOBS.md only documents the OpenClaw cron job; the system crontab entry is undocumented and violates the "all recurring tasks have been migrated to OpenClaw cron" guideline.
4. **Stale state**: OpenClaw cron job shows `runningAtMs` flag possibly stuck; its log file hasn't updated since 2026-03-02 22:36 UTC.

## Recommended Fix
- Migrate the **full pipeline** (data generation + git commit + push + Vercel deploy) into the OpenClaw cron job.
- Modify the OpenClaw cron job's payload to use `./scripts/generate-dashboard-data.sh` and then the deployment steps.
- Remove the system crontab entry for dashboard data updater.
- This centralizes control, adheres to best practices, and eliminates duplication.
- Optionally, adjust logging to capture output to `memory/dashboard-data.log`.

  3. Verify `data.json` format and monitor dashboard
