## Email Categorizer Systemd Service Fix (02:14–02:20 UTC)

**Issue**: Systemd user service `email-categorizer.service` was failing with `status=203/EXEC` repeatedly (restart counter >15100). The service tried to execute `/home/ubuntu/.openclaw/workspace/agents/email-sweep.sh` which does not exist.

**Root Cause**: Legacy systemd unit pointing to a non-existent script. The actual email sweep is properly handled by OpenClaw cron job `email-categorizer-cron` running the Python script `agents/email_sweep.py`.

**Actions Taken**:
- Disabled and stopped the failing systemd service: `systemctl --user disable --now email-categorizer.service`
- Removed the unit file: `/home/ubuntu/.config/systemd/user/email-categorizer.service`
- Reloaded systemd daemon
- Removed stale PID file `email-categorizer.pid`
- Verified cron-based sweep works by manual run: successfully fetched 100 unread emails and applied labels

**Result**: System noise eliminated; email sweep continues via cron as intended. No more 203/EXEC errors in syslog.
